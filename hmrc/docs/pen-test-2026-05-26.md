This report exists to back the **"Yes"** answer to the HMRC Developer Hub
question *"Has your application passed software penetration testing?"*.
HMRC explicitly accepts results from penetration test tools (not only
third-party suppliers), and the NCSC Penetration Testing Guide they
reference treats OWASP ZAP and testssl.sh as the canonical baseline tools.

passive/baseline scan; active scanning of a live multi-tenant database
risks data corruption and is reserved for a follow-up dedicated staging

## 4. Findings and remediation

### 4.1 High / critical

**None.** Zero ZAP `FAIL-NEW`. Zero testssl `CRITICAL` or `HIGH`. The
TLS configuration scored A+ before any work and the application surface
has no SQL injection, XSS, authentication bypass or session-management
findings.

### 4.2 Medium — remediated in this round

| # | Finding | Tool | Fix | Code |
|---|---|---|---|---|
| M1 | **`bp_csrf` and `bp_auth` cookies set without `Secure` flag** | ZAP 10010/10011, Observatory | Replaced `secure=IS_PRODUCTION` with `secure=_cookies_must_be_secure()` which defaults to True on any deploy that does not explicitly set `COOKIES_SECURE=0` for local dev. Root cause: Railway environment had no `ENVIRONMENT=production` env var so `IS_PRODUCTION` was False at runtime — a single missing env var silently downgraded cookie security. The new flag has the safe-by-default semantic that no env-var gap can recreate. | [`security_headers.py::cookies_must_be_secure`](../../security_headers.py), [`csrf.py`](../../csrf.py), [`core.py`](../../core.py) |
| M2 | **Strict-Transport-Security header missing** | ZAP 10035, Observatory | Added `max-age=63072000; includeSubDomains; preload` via the new `SecurityHeadersMiddleware`. Two-year max-age and `preload` together are the Chrome HSTS preload list's minimum. | [`security_headers.py`](../../security_headers.py) |
| M3 | **Content-Security-Policy header missing** | ZAP 10038, Observatory | Added an enforce-mode CSP. `default-src 'self'`, script-src allows our inline gtag bootstrap + `googletagmanager.com`, style-src allows the inline styles used across templates, `frame-ancestors 'none'`, `form-action 'self'`, `object-src 'none'`, `upgrade-insecure-requests`. | [`security_headers.py::CSP_VALUE`](../../security_headers.py) |
| M4 | **X-Frame-Options missing** (clickjacking) | ZAP 10020, Observatory | `X-Frame-Options: DENY` set as a separate header, plus `frame-ancestors 'none'` inside CSP for defence-in-depth. | [`security_headers.py`](../../security_headers.py) |
| M5 | **X-Content-Type-Options missing** | ZAP 10021, Observatory | `X-Content-Type-Options: nosniff` | [`security_headers.py`](../../security_headers.py) |
| M6 | **Referrer-Policy missing** | Observatory | `Referrer-Policy: strict-origin-when-cross-origin` — origin only on cross-origin navigations, no path/query leakage. | [`security_headers.py`](../../security_headers.py) |
| M7 | **Permissions-Policy missing** | ZAP 10063 | Locks down every powerful browser API we do not use — camera, microphone, geolocation, payment, USB, etc. | [`security_headers.py::PERMISSIONS_POLICY`](../../security_headers.py) |
| M8 | **Cross-Origin-Opener-Policy missing** | ZAP 90004 | `same-origin` — isolates our window from any opener cross-origin tab. | [`security_headers.py`](../../security_headers.py) |
| M9 | **Cross-Origin-Resource-Policy missing** | Observatory | `same-origin` — blocks other origins from embedding our assets. | [`security_headers.py`](../../security_headers.py) |
| M10 | **Sensitive information in URL on `GET /login`** — `password` query param renders in browser history, Referer and proxy logs even though the route never authenticates via the query string | ZAP 10024 | `SecurityHeadersMiddleware` redirects any `GET /login?...password=...` to bare `/login` so the credential never reaches the server access log or the browser history bar. The real login endpoint at `POST /api/login` was already POST-only. | [`security_headers.py::dispatch`](../../security_headers.py) |
| M11 | **Application error disclosure — 500 on five `/tools/bank-statement-for-*` pages** | ZAP 90022 | Triaged as a functional bug in the `tools/seo_page.html` template's handling of `None` for optional `data_requirements` / `compliance_section` / `challenges_section` on use-case pages. **Spawned as follow-up task** — not a security exploit (just discloses a Python exception in the body for these specific routes). Tracker: see the chip "Fix /tools/{slug} 500 on use_case pages". | tracked separately |

All M1–M10 code changes are covered by `tests/test_security_headers.py`
(21 tests, all green) and the full pytest suite (938 passed, 1 skipped,
1 xfail) is green against the patched tree.

### 4.3 Low — accepted with justification

| # | Finding | ZAP id | Decision |
|---|---|---|---|
| L1 | **Sub-Resource-Integrity attribute missing on `googletagmanager.com/gtag/js`** | 90003 | **Accepted.** Google rotates `gtag.js` continuously and explicitly does not publish or guarantee an SRI hash. Pinning a hash would silently break our analytics within days. Same trade-off used by Stripe.js across the industry. We will revisit if Google adds a stable subresource. |
| L2 | **Cookie-no-HttpOnly on `bp_csrf` (302 response)** | 10010 | **Accepted by design.** The double-submit-cookie CSRF pattern requires the JS to read `bp_csrf` and echo it in the `X-CSRF-Token` header — `HttpOnly` would defeat the entire scheme. The session cookie `bp_auth` IS HttpOnly. |
| L3 | **Cross-Domain JS Source File Inclusion on `googletagmanager.com`** | 10017 | **Accepted.** Expected — that is our analytics. CSP now constrains it to `script-src 'self' 'unsafe-inline' https://www.googletagmanager.com` only. |
| L4 | **`Cache-Control` directives on `/login`, `/forgot-password`, `/compare`** | 10015 | **Will fix in a follow-up.** These pages render no PII before login, but `Cache-Control: no-store` on the auth pages is best practice. Non-blocking. |
| L5 | **Information disclosure — suspicious comments** (5 pages) | 10027 | **Accepted.** Triaged — comments are routine template/SEO comments, no credentials or paths leaked. |
| L6 | **Modern Web Application / Authentication Request / Session Management Identified** | 10109, 10111, 10112 | **Informational only.** ZAP is flagging that we have a login form and a single-page-application surface — not a vulnerability. |
| L7 | **Non-storable content** (302, 404 responses) | 10049 | **Informational.** Response types ZAP cannot determine the cache-ability of. |
| L8 | **Cross-Origin-Embedder-Policy missing** | 90004 | **Deferred.** Enabling COEP requires every cross-origin resource we load to send `Cross-Origin-Resource-Policy: cross-origin` headers — `googletagmanager.com` does not. Enabling it now would break analytics. Tracked. |

### 4.4 TLS audit (testssl.sh) — A+ unchanged

```
Protocol Support (weighted)  100 (30)
Key Exchange     (weighted)   90 (27)
Cipher Strength  (weighted)   90 (36)
Final Score                   93
Overall Grade                 A+
```

- TLS 1.0, 1.1, SSLv3, SSLv2 — **not offered**
- TLS 1.2, 1.3 — offered
- ALPN — `h2, http/1.1`
- All offered cipher suites are forward-secret AEAD
- ChaCha20-Poly1305 and AES-GCM only — no CBC, no 3DES, no RC4
- Key exchanges: ECDH (P-256, P-384, secp521r1, X25519) + X25519MLKEM768 (post-quantum hybrid)
- HSTS now present (after this round of fixes)

No CVE-checked vulnerabilities triggered (Heartbleed, ROBOT, BREACH,
LUCKY13, BEAST, CRIME, POODLE, etc. — all clean).
